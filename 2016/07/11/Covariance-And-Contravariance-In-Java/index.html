Java中的协变与逆变

发表于 2016-07-11 | 更新于 2017-09-17 | 分类于 JavaSE | 阅读次数
字数统计 3,667 | 阅读时长 13

　　“协变”一词我并不是第一次见到了，在以前我学习C++的时候就碰到过，而到后来学习Java的时候也遇到了，而在学习Java泛型时更是碰到“逆变”一词。不过我见得最多的是“协变返回类型”，想必大多数人也跟我差不多。“协变”、“逆变”这些概念性的名词看起来就十分高大上的样子，但是很多基础书籍并没有明确地说明这些概念到底是什么意思，更不用说什么长篇大论描述了。所以，很多时候看到这些概念我都是似懂非懂，而今天自己就查阅了各种资料并通过这篇博文总结一下。

里氏替换原则（Liskov Substitution Principle, LSP）

在讲述“协变”、“逆变”概念之前，首先要说一个面向对象设计的基本原则——里氏替换原则。

　　我们要深入了解那些概念，那首先就得认识这一个里氏替换原则，“协变”、“逆变”这些概念估计就是从这个原则上推导出来的（虽然我没查过是不是，但从感官上来说确实像这么回事。）只要我们知道这个基本原则是什么，那么在理解之后要详述的概念是事半功倍，所以，我并不急着一上来就扯一大堆关于“协变”和“逆变”的内容。

里氏代换原则（Liskov Substitution Principle, LSP）的通俗定义

  • Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it.
    所有引用父类型的地方必须能透明地使用其子类型的对象。

　　这定义十分好理解吧？更通俗直白地讲，就是某段程序代码所使用的是父类型，而无须改变这段程序代码就能将其子类型替换进去进行使用。还不明白？那你可以想象一下平时自己在编程的时候是怎么做的，那你立刻就能理解这句话的实际含义。比如我是使用Java进行编程的。我定义了接口和抽象类，并在某些代码中使用了这些接口类型或抽象类类型的对象，而程序代码的具体实现就是使用接口的具体实现类对象或者抽象类的子类对象，这就是平常我们都在运用的多态啊。

　　我查过了关于该原则的定义，各式各样的描述都有，但我觉得上面这个是最容易让人理解的，至于更正式更规范的定义，如果你想知道可以自己Google百度搜索一下，这里就不再赘述了。

协变（Covariance）与逆变（Contravariance）

　　关于这些概念的解释，我查阅了很多文章，描述不一，但它们所要表达的意思似乎都是一样的，在这里我就总结一下。如果你想要查看这些文章，打开本文最底下所提供的参考资料链接即可。下面就开始详述我们最为关心的内容了。

概念描述

概念是十分抽象的，在什么都不了解的前提下，这些概念对于你我来说可以说是不知所云或者似懂非懂，反正就是一脸懵逼，完全找不到切入角度。

文字定义

  • 协变（covariance）与逆变（contravariance）统称为变体或者变体或变型（variance）。

  • 变型（variance）描述具有子类型关系之间的类型经过“类型转换”（type transformation）后，所构造出更复杂的类型之间的关系是如何变化的。

    此处所说的“类型转换”非平日我们经常所提到的诸如隐式类型转换（implicit type conversion）、显示类型转换（explicit type conversion）、强制类型转换（type coertion），
    这些都统称为类型转换（type conversion），描述的是数值类型以及父子类型的转换，比如int转换为long、String转换为Object。

  • 协变（covariance）指的是具有子类型关系之间的类型经过“类型转换”后，所构造出更复杂的类型之间仍保持着子类型关系。

    协变更广泛的含义：一种事务变换为另一种事物后，仍保持着相同的形式或规律，这就是协变。

  • 逆变（contravariance）与协变相对立，指的是具有子类型关系之间的类型经过“类型转换”后，所构造的更复杂的类型之间建立了逆向子类型关系。

  • 不可变（invariance） 表示协变与逆变2种情况同时不成立。

看完上面这些文字，如果你还是摸不着头脑，没关系，你只需要知道它们都是描述类型规则的。
但你想更深入地理解，则必须要抓住其中的重点词“具有父子关系之间的类型”、“类型转换”来继续看以下的内容。

tips：本文所说的“类型转换”指的是从一种类型构造为另一种新的类型，是质的转变，即新类型与原类型不在同一个抽象层面上。

具体公式

假设此处有X、Y2种类型，而符号≤表示子类型关系（比如：X ≤ Y即类型X是类型Y的子类型），并且f表示类型转换，
经过类型转换后便构造出更复杂的类型f(X)和f(Y)，那么，就可以得出以下这些结论：

  • 如果X ≤ Y，经过f后有f(X) ≤ f(Y)，那么f具有协变性。
  • 如果X ≤ Y，经过f后有f(Y) ≤ f(X)，那么f具有逆变性。
  • 如果以上两者都不符合，那么类型转换f具有不可变性。

公式还是抽象的，但在这里我就给出一个最简单的例子你就能马上理解这些公式了。

前提条件

问题详细分析

从之前所看到的前提条件我们可以确定2件事：

  • 从fruits 协变数组中读取出来的元素是Fruit类型。
  • 在编译期，编译器是绝对允许我们将Fruit以及其子类型的元素写入到fruits协变数组中。

从协变数组fruits中读取元素是完全安全的，无论是编译期还是运行时，都不会发生任何问题，正如上面所说的，fruits所引用的数组其实是Apple[]类型，从Apple[]类型数组中读取出的元素绝对是Fruit类型啊。

但是将Fruit类型以及其子类型的元素写入到协变数组fruits中是有可能在运行时出现问题的；正如前面所讲：编译器是完全没有理由阻止我们将Fruit类型以及其子类型的元素写入到协变数组fruits中，因为这个操作完全可以通过编译期的类型检查。

但是你可以试想一下：如果这个协变数组是从别人所写API的方法中返回的，对于方法用户的我们而言，可能忽视了这个协变数组的真实类型，又或者是我们写代码有时写得头昏脑胀，完全忘记这个数组是一个协变数组，我们很自然地把它当作一个普通数组来用；在这2种情况下我们都很自然把某些符合父子关系类型的元素写入到该数组中，编译期完全不会报错，然而到了运行时，我们很可能就会收到一个异常java.lang.ArrayStoreException。

协变数组的结论

这是Java协变数组的“缺陷”，但我们其实十分清楚引发这个问题深层原因，所以我们在利用数组的协变特性时可以完全规避这个问题：
在利用数组的协变性时，应该尽量把协变数组当作只读数组使用。

方法

在Java 1.5以及之后就支持协变返回类型（covariant return type）这一个特性：
这特性指的是子类型覆盖父类型方法时，子类型方法的返回类型是父类型方法返回类型的子类型，我们就把这个返回类型称作协变返回类型。

或许你很懵逼，这方法返回类型也具有协变性？
正如前面概念所说的那样，在这里对应的过程就是简单类型经过“类型转换”后所构造出的复杂类型正是方法返回类型。

有了具体代码示例就更好理解了。Base和Derived之间具有父子类型关系，然后经过“类型转换”后，Base成为了Super类get方法的返回类型，而Sub继承Super覆盖它的get方法时，Derived成为了它的返回值；即Base和Derived成为了抽象层面更高的方法返回类型，但它们仍保持着一种子类型关系，所以说方法返回类型具有协变性。

但这种特性在Java 1.5之前的版本就不适用了，以前子类型覆盖父类型的方法，必须是方法签名和方法返回类型都保持一致的。

在Java中关于方法的协变逆变就只有这么多内容。

泛型

Java泛型是具有不可变性。但尽管如此，Java泛型自带的特性之一可以使得泛型具有协变性或不可变性，在此处就不作详细讨论，在另一篇探讨Java泛型的博文里再作详细分析。
